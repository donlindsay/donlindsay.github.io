Facts Come From the Graph, Not the Prompt

There is a quiet decision hiding inside every generated devlog and every page on this site: where did the facts come from? You can paste them into the prompt, or you can retrieve them from the knowledge graph. The two produce text that looks identical — and they are not the same thing at all.

Two ways to write the same sentence


PROMPT-FED  ·  composition
   human curates facts  ->  prompt  ->  LLM composes  ->  devlog / page
   x  the graph sits unused          x  the claims are unverifiable

GRAPH-GROUNDED  ·  retrieval
   AllegroGraph  ->  SPARQL retrieves  ->  LLM writes from rows  ->  devlog / page
   +  every claim traces to a triple   +  faithfulness scored against the graph

Why the prompt is a trap

When you hand-paste facts into the prompt, four things happen, none of them good:

  • The facts are a snapshot, curated by whoever wrote the prompt. They can be stale, partial, or wrong — and the model has no way to know. It will render your mistake fluently and with confidence.
  • You are testing prose, not truth. The model's job collapses to "make these sentences flow." Whether they are accurate was decided before the model ever ran.
  • There is nothing to check the output against. The "source" was just more prose; a claim cannot be verified against a source that does not outlive the prompt.
  • The expensive thing you built sits unused. You stood up a provenance store with a chain of custody, then routed the facts around it.

Why retrieval is the real thing

Grounding in the graph inverts all four:

  • The facts are the current state, not a memory of it. If the council seat roster or the dataset catalog changes, the next devlog changes with it — nobody re-curates a fact-set.
  • Faithfulness becomes checkable. Every sentence should trace to a triple that was retrieved; you can score hallucination by flagging any claim that names something the graph never returned.
  • Writing and publishing share one source. It is the same pipeline this site already uses — build_site.py SPARQL-queries a SKOS taxonomy in site.kg to lay out these very pages. The prose and the published structure agree because they drink from the same well.
  • It tests the capability that matters — can the model query the system of record and stay faithful to what it returned? That is the agentic loop: retrieve, then write.

The deeper principle

This is the same stance the stack already takes internally. data.facts is durable and authoritative; mem.agent self-poisons, because the agent re-ingests its own paraphrases until recall drifts. Authoritative truth lives in the durable, queryable graph — not in a context window, not in the model's memory of what it once said, and not in a prompt typed by hand.

So "facts come from the graph, not the prompt" is really a provenance stance. The prompt is ephemeral, unauditable, and owned by whoever typed it. The graph is durable, queryable, and carries custody. For a stack whose entire thesis is that the knowledge graph is the system of record, letting facts originate in the prompt quietly defeats the point — it launders unsourced claims into authoritative-looking output.

How this is enforced here

The writing battery that grades the local models was rewritten to obey this principle. It no longer feeds the model a hardcoded fact-set; instead it:

  • retrieves the council seat roster (?seat shell:invokesModel ?model) and the dataset catalog (dcat:dataset + dcterms:description) from AllegroGraph;
  • has the model write the devlog, overview, and index blurb from the retrieved rows;
  • scores coverage of graph-true facts and flags fabrications — any plausible name the graph never returned;
  • and in its end-to-end task, makes the model write its own SPARQL, runs it against the graph, and feeds the rows back to be written up.

The grade moved from "is this fluent?" to "is this true to the graph?" — the only question worth asking of a system that has a graph.
