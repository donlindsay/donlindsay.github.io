DEVLOG SNAPSHOT — AI Stack — 2026-06-11 (NVIDIA agent stack + inference escalation)

This snapshot covers making the 196 installed NVIDIA skills consumable by the graph-native agent, a two-tier inference escalation router, the NVIDIA agent-stack install onto aishell, and the transport bug that blocked NemoClaw's runtime.

1. This session

  • NVIDIA skills made graph-consumable. aistack.skills_catalog parses the installed SKILL.md frontmatter into an RDF index (nvskills.ttl loaded as the vocab.kg named graph <http://kg.local/nvskills>): 196 nvsk:Skill across 28 families, SPARQL-discoverable by description and family. The cuOpt family's full bodies were RAG-grounded into data.docs via aistack.ground_skills so they retrieve from the vector store; discovery is wired into the operating note as shell:SkillDiscoverNvSkills.
  • Two-tier inference escalation router. aistack.nemorouter + nemorouter_server expose ONE OpenAI-compatible endpoint where the model field (nemotron|council|auto) selects the tier — Tier 1 local Nemotron-4B (free) and Tier 2 the Cloud Council. It reuses the existing SHELL escalation machinery rather than inventing a new protocol: ACCESS->NET=CLOUD opt-in (config NET_CLOUD), CALLER_BUDGETS, spend metered as caller 'nemoclaw-agent', and triage.classify for auto routing. Cloud-off or over-budget degrades gracefully to Nemotron (shell:ProtoNemoClawInference).
  • NVIDIA agent stack installed on aishell. OpenShell (uv tool) and NemoClaw (its real installer, run under a Bash permission rule); aishell added to the docker group with GPU-in-container verified (RTX 5070). Added two GPU venvs — cuda-python 13.3.1 and nvidia-resiliency-ext 0.6.0 on torch 2.12+cu13 — both GPU-verified. NemoClaw's own nim-local provider failed because it can't see the WSL GPU (it lives at /dev/dxg, not /dev/nvidia*), so inference was onboarded against the nemorouter instead, validated, and baked into the sandbox image.
  • Hard wall: OpenShell<->WSL2 transport bug. Short gRPC calls succeed, but the long-lived gateway<->sandbox supervisor stream dies at ~215s with an h2 protocol error and the SSH-over-gRPC dashboard forward exits 255, tearing down the sandbox. Root-caused away from the obvious suspects: br_netfilter is loaded, MTU is a uniform 1500, and the failing traffic is docker-internal. WSL networkingMode=mirrored produced an identical failure (ruling out the WSL NAT layer) and was reverted — this is an upstream OpenShell/NemoClaw alpha bug.

2. Lessons / TODO

  • Cataloguing SKILL.md frontmatter into RDF and RAG-grounding the bodies is the right way to make external coding-agent skills consumable by a graph-native agent: discover via SPARQL, retrieve via the vector store.
  • The escalation router reuses the stack's existing local<->cloud protocol — one endpoint, model field selects the tier — instead of a parallel mechanism.
  • Mirrored networking does not fix OpenShell's long-lived gateway<->sandbox streams because that traffic is docker-internal, not WSL<->Windows; report upstream or run NemoClaw off WSL.
  • TODO: make the nemorouter persistent (add to start-stack.sh); reconcile the OpenShell version skew (0.0.59 uv tool vs 0.0.44 NemoClaw-bundled).
