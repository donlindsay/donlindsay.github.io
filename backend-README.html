AI Stack Console (main)

One backend, three frontends. The agent, skills, knowledge-graph helpers and AllegroGraph-backed memory live once in the aistack package; the desktop (Qt), web (Gradio) and cli (terminal) frontends are thin shells over it.


./run.sh                 # desktop — native Qt window (default)
./run.sh web             # Gradio  → http://localhost:7860
./run.sh cli             # terminal REPL
./run.sh cli "how many documents mention OpenAI?"   # one-shot
./run.sh api             # FastAPI → http://127.0.0.1:8765  (for native front ends)

run.sh forces QT_QPA_PLATFORM=xcb (Qt6-Wayland under WSLg segfaults; X11 is stable). Under the hood it just runs venv/bin/python main.py <frontend>.

Layout


main.py                 # entry point: dispatches to a frontend (lazy import)
api.py                  # FastAPI shim over the backend — for native front ends
aistack/                # the shared backend — single source of truth
  config.py             #   endpoints, repos, the model
  core.py               #   HTTP, Ollama, AllegroGraph helpers, service health
  skills.py             #   the 13 agent skills + tool schema + NL→SPARQL + charts/graphs
  agent.py              #   the tool-calling loop + AllegroGraph-backed memory
  ops.py                #   richer page operations for the form UIs
frontends/
  desktop.py            # PySide6/Qt: frameless dark skin, sidebar, 6 pages, status bar
  web.py                # Gradio: 6 tabs
  cli.py                # terminal: truecolor REPL (slash-commands) or one-shot

Only the chosen toolkit is imported, so cli needs neither Qt nor Gradio.

Native front ends (the API)

main.py api serves the backend over plain HTTP at http://127.0.0.1:8765 so a native Windows UI (Avalonia/WPF/Tauri/anything) can drive the same agent and skills — the backend stays Python, the UI is any language. WSL2 forwards localhost, so a Windows client reaches it with no extra setup. start-stack.sh brings it up as systemd: ai-api.

The agent

The agent runs on a free engine × model matrix (the pedalboard's Reaper, slots 0 + 1): Engine ∈ {Ollama, llama.cpp} × Model ∈ {Nemotron, Twin-Llama}, selectable independently. The default is llama.cpp × Nemotron (nemotron-3-nano:4b, served on the GPU by llama-server built for CUDA sm_120). localllm._resolve() maps each pair to the live backend with a graceful Ollama fallback; the single-resident 12 GB GPU is governed by Reaper (engaging a GPU engine evicts the others). vLLM was the prior GPU engine and is now shelved (dormant; swap-in-able via set_engine("vllm")).

The loop is a single bounded tool-calling loop (Ollama /api/chat, or the engine's OpenAI API for llama.cpp/vLLM). There is no router — the SYSTEM prompt handles restraint, so the model answers general questions directly and only calls a tool when it genuinely helps.

Capabilities are wired in as 13 skills:

web_search · knowledge_base · extract_url · generate_image · sparql_query · graph_query (NL→SPARQL) · ask_table (tables) · chart (bar/line/pie) · graph_image (subgraph picture) · ask_claude / perplexity_search (cloud) · save_fact · forget

Image-producing skills (generate_image, chart, graph_image) render inline in the desktop chat, scaled to the box.

Memory is AllegroGraph-backed: each exchange is embedded (nomic-embed-text) into the agent_memory repo and semantically recalled before each reply (toggle per frontend). save_fact promotes durable facts to RDF triples in kg_facts; forget deletes matches.

NLQ implements AllegroGraph's native-NLQ design (schema hint + few-shot NL↔SPARQL pairs) scriptably — AG 9.0.1 exposes no programmatic NLQ API. nemotron emits a single SPARQL query that runs against the chosen repo.

CLI commands

/help /status /memory on|off /repo [name] /clear /rag <q> /search <q> /nlq <q> /sparql <repo> <q> /extract <url>… /quit — anything else goes to the agent.

Notes

  • No CORS — runs on the stack host and calls the n8n webhooks + AllegroGraph directly.
  • Routerless, single-model — nemotron does tool selection itself; the SYSTEM prompt

steers restraint (answer directly when no tool is needed) and synthesis (use tool results in one or two tight sentences). A shared IDENTITY keeps one consistent voice.

  • AllegroGraph LLM predicates — askMyDocuments (documents + citations) and

askForTable (tables) run natively on nemotron, the only local model that passes Franz's function-call parser for both (others emit prose → car of "" not listp). ask_table keeps a portable Ollama→JSON fallback for consistent columns.

  • Cloud skills (ask_claude, perplexity_search) send data off-box; everything

else is local-only.

  • Desktop UI — frameless dark window with a custom title bar (Wayland-under-WSLg

segfaults, so X11/xcb), a UI-scale slider, and the horse app icon. The status bar polls all five services every 10 s.

Supersedes the standalone ../legacy/console (web) and ../legacy/desktop (Qt) apps, whose backends were duplicated. Part of the local AI stack — see ../README.md, ../GUIDE.md.
