DEVLOG — 2026-06-08 — Shelve Ollama + the SHELL pedal

id: urn:kg:doc:devlog-2026-06-08-shell · source: devlog · date: 2026-06-08

Two threads landed this session: Ollama fully removed from the runtime, and a new SHELL pedal that revives agentic tool-use.

1. Ollama shelved → embeddings on llama.cpp

Ollama was kept only to serve nomic-embed-text to AllegroGraph's vector store. Research into AG 9.0.1 showed its embedder is demo|openai|ollama with no custom-endpoint for openai, but the ollama embedder's host/port IS configurable (agraph.cfg <llm ollama> → host 127.0.0.1 port 11434, and AG calls GET /api/tags + POST /api/embed).

So instead of moving vectors out of AG, we kept them in AG and replaced the Ollama daemon with a drop-in embed adapter at :11434:

  • /opt/aistack/embed-adapter/app.py — stdlib HTTP, loads a nomic-embed-text-v1.5 GGUF in-process via llama-cpp-python, answers /api/tags + /api/embed (768-d, L2-normalized).
  • data.docs-vec + mem.agent-vec re-indexed through it; rag.py/memory.py unchanged.
  • localllm.py: _ENGINES=("llamacpp",); default + graceful fallback → llamacpp; Ollama/vLLM flagged shelved; _ollama_daemon_up() is a process check (the :11434 HTTP probe now hits the adapter).
  • start-stack.sh launches the adapter in place of ollama serve.

Verified live: RAG retrieval scores 0.67 on-topic, rag.ask cites correctly, memory.recall returns hits, /status shows ollama:false, /local-matrix is llamacpp-only, pgrep ollama empty. Single runtime (llama.cpp) for chat and embeddings.

2. SHELL pedal — agentic chat-editor (CODE slot, was Tabby)

A bespoke chat-editor that turns the board into an agentic coding/shell console — closing the two gaps from the fitness report (dormant Nemotron agentic capability + narrow llama-cpp-python use).

  • aistack/shell.py — agent() loop drives the engaged llama.cpp model with structured JSON output to pick and run shell commands, observe, and answer. classify() permission policy (read-only auto / mutating confirm / destructive deny, strictest segment wins); executors for Ubuntu bash + Windows PowerShell (per the ACCESS->SHELL knob); provenance to custody (PROV).
  • api.py — /shell/{chat,exec,classify,toolcheck}.
  • App — slot-6 pedal Tabby → SHELL; MAIN page is the chat-editor (transcript + input + Run), wired to the ACCESS->SHELL knob via ApiClient.ShellChatAsync. Builds clean (Release+Debug).

Verified: classify(rm -rf /)=deny, classify(sudo …)=deny; the agent ran find … | wc -l and answered correctly; the live /shell/chat endpoint drives the loop end-to-end.

3. Opt-in Opus 4.8 engine

SHELL can be driven off-box by Opus 4.8 when ACCESS->WEB=CLOUD (else it stays on the local Nemotron). core.anthropic_messages() wraps the Anthropic Messages API with native tool-use (tool_choice:any, disable_parallel_tool_use); the key lives only in /opt/aistack/.anthropic-key (chmod 600), never in git. The two engines now share one driver — same permission policy, same custody provenance — only the model differs.

4. Typed file tools + approve-resume

The agent went from "runs commands" to edit-capable, with a human gate on every write:

  • Tools alongside run_command: read_file (paged, numbered lines → classifies auto), write_file and apply_patch (exact-unique replace → classify confirm). Secret paths (.ssh/id_, .aws/credentials, /etc/shadow, *-key) and system-path writes (/etc, /boot, /usr, …) are deny.
  • A unified resumable driver (_drive/_model_turn/_append_result) replaces the two old per-engine loops. A mutating op pauses the loop and returns {gate, pending_id} instead of dropping the turn; resume(pending_id, approve) runs (or skips) the op and continues. New POST /shell/approve; the app renders an inline Approve / Deny chip.

Verified: deterministic gate/approve/deny/expired unit test; live HTTP e2e on Nemotron — autonomous read with no gate; write gated → approve → file written → final.

5. Durable IDE sessions in AllegroGraph

SHELL is an IDE, so a session must survive a restart. New repo shell.sessions (aistack/shellstore.py):

  • Each session persists its transcript + working context (shell/cwd/engine) + open gate. The conversation blob is base64-encoded into the sh:state literal — arbitrary file/tool text becomes pure [A-Za-z0-9+/=], sidestepping SPARQL-escaping (read-heavy transcripts had triggered AG 400s with hand-escaping). Queryable mirror triples (id, updated, status, engine, pending, title) drive the session list and let resume() find an open gate by pending-id after a cold start.
  • agent(session_id=…) continues a durable session — real multi-turn memory, which SHELL had none of before. resume() is restart-durable: on an in-memory _PENDING miss it reloads from AG by pending-id. Best-effort: AG down → in-memory-only (the pre-IDE behavior).
  • API: /shell/chat takes+returns session_id; GET /shell/sessions, GET/DELETE /shell/session/{id}. App persists its session id to %APPDATA%\aistack-shell-session.txt and reloads the transcript on reopen; a New button starts fresh.

Verified: deterministic restart-sim (clear _PENDING → resume via AG → file written → status idle → transcript grows); live HTTP with a real ai-api restart between gate and approve — the file was written across the restart, and a same-session follow-up returned READY.
