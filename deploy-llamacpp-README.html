llama.cpp serving — PRIMARY Slot-0 engine

OpenAI-compatible llama-server on 127.0.0.1:8002, served-model-name nemotron | twin-llama. This is the default GPU engine (default quadrant llama.cpp × Nemotron); vLLM is shelved (../vllm/).

  • serve.sh — runtime launcher (lives at /opt/aistack/llamacpp-worker/serve.sh).

Reads .target-model (nemotron | twin-llama) and loads the matching GGUF from /opt/aistack/models/gguf/ with full GPU offload (-ngl 999 --ctx-size 8192 --jinja). The GGUFs are the exact Ollama blobs (parity with the ollama row), copied once.

  • ai-llamacpp.service — systemd unit (installed to /etc/systemd/system).

Not enabled at boot. localllm starts/stops it on engage (single-resident GPU).

Build (CUDA sm_120 / Blackwell, RTX 5070)

Built from source at /opt/aistack/llama.cpp:


cmake -B build -DGGML_CUDA=ON -DCMAKE_CUDA_ARCHITECTURES=120 -DLLAMA_CURL=OFF -DCMAKE_BUILD_TYPE=Release
cmake --build build --target llama-server -j"$(nproc)"

system_info confirms CUDA : ARCHS = 1200 | BLACKWELL_NATIVE_FP4 = 1. ~180 tok/s on nemotron. (Note: this build made --flash-attn an optional-arg flag — [on|off|auto], default auto — so serve.sh omits it rather than risk it eating the next token.)

Model-swap (one model per process on 12 GB)

localllm._sync_llamacpp writes the family to .target-model and (re)starts ai-llamacpp; _llamacpp_loaded_model() reads /v1/models to confirm. Single-resident GPU: engaging llama.cpp evicts Ollama + stops vLLM.

Tool-calling

Currently OFF (config.LLAMACPP_TOOL_MODELS=set()) → clean chat + RAG. Enable later via the GGUF's tool chat template (--jinja) or a GBNF grammar, verify with diag.py, then populate the set.

Gotcha

Editing serve.sh through the Windows \\wsl.localhost\… path strips the +x bit → systemd 203/EXEC. chmod +x after any such edit.
