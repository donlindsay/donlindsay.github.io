1. Opening statement

Concur with the Reporting Senior's central finding: "AI-Stack + AllegroGraph = neuro-symbolic / KG platform" is the right center of gravity, and AllegroGraph should be treated as a first-class reasoning and memory substrate rather than a bolt-on datastore. The correction is granted in full — with one turn. The substrate work is already done: AllegroGraph is wired through retrieval, memory, sessions, provenance, querying, and visualization. The gap is not architecture; it is that the docs undersell it, reading like generic MLOps when the running system is a working neuro-symbolic platform. Here, neuro-symbolic = Nemotron / llama.cpp (neural) ⨉ AllegroGraph (symbolic: RDF, SPARQL, the in-store Prolog reasoner, and vectors-in-store).

2. Graded marks

  • (a) AllegroGraph as first-class reasoning + MEMORY substrate — ★★★★★

GraphRAG retrieval runs over AG's native vector store across the document corpus (data.docs); long-term agent memory and durable facts live in AG (mem.agent, data.facts); every SHELL session's transcript + context is persisted as RDF in shell.sessions and survives restarts.

  • (b) Neuro-symbolic workflows — ★★★★★

Neural embeddings from llama.cpp (nomic-embed-text) and Nemotron generation are joined to symbolic RDF/SPARQL and the in-store Prolog reasoner; custody is a tamper-evident chain of cc:Event triples (cc:seq + cc:prev) in the custody artifact graph.

  • (c) NL→SPARQL Graph pedal + KG/Gruff visualization — ★★★★☆

The Graph pedal compiles English into generated SPARQL and executes it against AllegroGraph; the KG / Gruff pedal renders knowledge-graph snapshots.

  • (d) Docs positioning — ★★☆☆☆ (the real finding / action item)

Documentation foregrounds generic MLOps and budget/cloud tone instead of the KG substrate. This is the Reporting Senior's true deficiency and the focus of the development plan below.

3. Corrective actions / development plan

Adopt the reviewer's doc subtree and make the Knowledge-Graph section top-level. Add a conceptual paragraph naming AllegroGraph and a block diagram with AG as the knowledge layer:


  [ llama.cpp / Nemotron ]  --neural-->  [ Pedals: Ask · Graph · KG/Gruff · SHELL ]
                                              |  SPARQL / vectors / Prolog
                                              v
   ====================  AllegroGraph (knowledge layer)  ====================
   data.docs (vectors) · mem.agent · data.facts · shell.sessions · custody · site.kg

Doc tree: Concepts (RDF / repositories / auth) → Setting up a repository → Loading data (bulk + streaming) → Querying with SPARQL (2–3 examples) → Using AG from agents/LLMs:

  • Recipe 1 — SPARQL → JSON results → feed to llama.cpp/Nemotron as grounded context.
  • Recipe 2 — NL → generated SPARQL (Graph pedal) → answer → record to custody.

Then Operating notes. Move governance/budget/cloud tone to an appendix. Replace generic TensorFlow examples with AG-centric ones (vector search over data.docs, SKOS traversal of the site IA, custody-chain queries). Note: the doc-site's own information architecture is a SKOS ConceptScheme in site.kg, queried by the generator (build_site.py) via SPARQL — the docs already eat their own dog food.

4. Answer to the closing question

For THIS stack, AllegroGraph is primarily RAG-style QA over the corpus (the Ask pedal, vectors-in-store). But it is also a native entity-event model: custody/provenance (cc:Event chains) and durable SHELL sessions ARE entity-event graphs. And it carries light analytics/BI via SPARQL aggregates in the Graph pedal. So: RAG-first, entity-event by construction, analytics as a bonus.
