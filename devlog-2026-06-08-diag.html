Devlog — 2026-06-08 — Re-battery, tool-call/RAG diagnostic, Nemotron parser fix

Session arc across the four Reaper quadrants (Engine × Model = Ollama/vLLM × Nemotron/Twin-Llama): finalize the Llama-Nemotron swap, re-run the competition battery fairly, build a stage-isolating diagnostic, and use it to find + fix a masked tool-calling bug.

1. Llama-Nemotron adoption finalized (8173bf6)

vLLM "Nemotron" = nvidia/Llama-3.1-Nemotron-Nano-4B-v1.1 (pure Transformer, ~56 tok/s; util 0.90, max-len 5120, FLASH_ATTN). Reasoning-first, so the agent prepends NVIDIA's "detailed thinking off" and strips <think> → ~0.9 s clean answers.

2. Shared thinking-off fix + re-battery (d6fbdb2, 961d0fd)

  • Moved "detailed thinking off" + <think> strip into localllm.generate/bench (one place) so

rag.ask and the battery get fast clean answers, not just run_agent. Verified: vLLM×Nemotron generate 1.42 s clean, bench 58.7 tok/s.

  • battery.py gained Phase C — per-quadrant real run_agent tool-loop routing table.
  • Result: vLLM×Nemotron a co-winner (81 pts, 136.8 tok/s) tied with Ollama×Nemotron; coverage 6/6.
  • A spawned task chip ("Enable tool-calling for vLLM Llama-Nemotron") was run by the user; its work

(an llama_nemotron_json parser plugin + serve.sh wiring + config re-enable) was finalized and the battery appeared to show vLLM×Nemotron driving tools.

3. Tool-call + RAG diagnostic (diag.py, e744fe9)

Built a diagnostic battery distinct from the competition battery: discrete pass/fail probes per stage, per quadrant, with payloads. Tool-call: A0 direct parser probe (engine API, one tool — isolates the parser from the agent loop) → selection → args/use → restraint → output hygiene. RAG: retrieval relevance → distractor precision → ranking → read-after-write → grounded generation, on a seeded data.diag repo (1 target + 3 distractors). Honest about fallback: a vLLM quadrant not natively serving its model is marked ⚠️ NOT TESTED, never a silent pass off the Ollama fallback.

KEY FINDING — vLLM×Nemotron tool-calling was BROKEN

The llama_nemotron_json plugin 400'd on EVERY tool request: LlamaNemotronJSONToolParser.__init__(self, tokenizer) vs vLLM 0.22.1's base ToolParser.__init__(self, tokenizer, tools=None) (instantiated with both positionally → "takes 2 positional arguments but 3 were given"). run_agent then returned (no response). This corrected commit d6fbdb2's "verified working" claim — that was the pre-restart llama3_json parser or the silent Ollama fallback, exactly the blind spot the diagnostic removes. Lesson: "tools fired in the battery" ≠ "this engine tool-called" — always check the backend that actually ran.

4. Parser fix + disable (a4e1b3f)

  • Patched the plugin __init__(self, tokenizer, tools=None) → super().__init__(tokenizer, tools).

Parser now fires (diag A0 ✅, 400 gone). Mirrored to deploy/vllm/nemotron-tools/.

  • But the 4B model does NOT reliably emit <TOOLCALL>[…]</TOOLCALL> JSON in the full multi-tool agent

loop — it leaks [TOOLCALL: name?args] as content. So nemotron was removed from VLLM_TOOL_MODELS: vLLM×Nemotron does clean direct chat + RAG (both verified) instead of leaking tool markup. ai-api reloaded (transient-unit recreate) so the live default reflects it.

  • Follow-up (task chip): re-enable once the chat-template/prompt is tuned for the agent loop.

Other diagnostic findings
