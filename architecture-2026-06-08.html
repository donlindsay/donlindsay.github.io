AI Stack — current architecture (2026-06-08)

id: urn:kg:doc:architecture-2026-06-08 · source: audit · date: 2026-06-08

AI Stack current architecture (2026-06-08). The local LLM runs on the Reaper engine x model matrix. The SOLE live engine is llama.cpp (llama-server, OpenAI-compatible API on port 8002, built for CUDA sm_120 / Blackwell); the default engaged quadrant is llama.cpp x Nemotron (nemotron-3-nano:4b), with Twin-Llama as the second model. Ollama, vLLM, n8n and ComfyUI are now OPTIONAL ADD-INS (shelved/dormant): out of the active engine matrix, services stopped, code preserved + swap-in-able.

AllegroGraph 9.0.1 is the knowledge graph + RAG store. Its native vector stores (data.docs-vec for RAG, mem.agent-vec for agent memory) embed via nomic-embed-text — formerly served by the Ollama daemon, now by an in-process llama.cpp embed adapter (llama-cpp-python + a nomic GGUF) that speaks the Ollama embeddings API on :11434, where AG's <llm ollama> embedder points (host/port configurable in agraph.cfg). So the stack runs on a SINGLE runtime — llama.cpp — for both chat and embeddings, with zero Ollama binary; vectors stay in AllegroGraph and rag.py/memory.py are unchanged. The 12 GB GPU (RTX 5070) is single-resident. Stable Diffusion runs as a socket-activated diffusers worker.

New: the SHELL pedal (CODE slot) — a bespoke agentic chat-editor that turns the board into a coding/shell IDE. One engine-agnostic driver runs a tool loop over run_command plus typed file tools (read_file with paging, write_file, apply_patch) under a permission policy: read-only ops run automatically (auto), mutating ops (writes/edits/deletes) pause on a per-op confirm gate, and destructive/secret-touching ops are denied (secret paths like .ssh/id_, .aws/credentials, *-key and system-path writes are hard-denied). Every executed op's provenance is written to AllegroGraph custody. The default engine is the on-box llama.cpp model (Nemotron, structured-JSON tool calls — reviving the agentic tool-use that had been deferred on llama.cpp); flipping ACCESS->WEB=CLOUD opts into Opus 4.8 via the Anthropic Messages API (native tool-use), with identical policy + provenance.

Because SHELL is an IDE surface, sessions are durable: the full transcript, working context (shell/cwd/engine), and any open confirm-gate are persisted to a dedicated AllegroGraph repo (shell.sessions) — the conversation blob base64-encoded into one literal, with queryable mirror triples (id, updated, status, engine, pending, title). A confirm gate returns a pending_id; approving it continues the same loop, and because the gate is persisted, an Approve survives a full ai-api restart (resume() reloads the session from AllegroGraph by pending-id). The app remembers its session id locally and reloads the transcript on reopen. Exposed over /shell/{chat,exec,classify,toolcheck,approve}, GET /shell/sessions, and GET/DELETE /shell/session/{id}.

Self-building (Part B): the same edit-capable agent can operate on its OWN source (/opt/aistack/main) via /shell/selfbuild — propose-only (reads + a precise old→new proposal, no writes) or apply (minimal apply_patch → py_compile verify → git add staged for a human-approved commit). Self-modification rides extra guardrails: protected infra paths (start-stack.sh, agraph.cfg, *.service, engine serve.sh, the venv, secrets) are write-denied; destructive/history-rewriting git (push, reset --hard, clean, checkout --, restore, rebase) is denied; read-only git runs un-gated; source edits and git add/commit are confirm-gated, so a human approves every change and the agent never pushes. Each self-edit is written to prov.custody — explainable self-modification.
