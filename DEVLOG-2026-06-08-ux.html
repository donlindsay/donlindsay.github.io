DEVLOG — AI Stack — Pedalboard UX + Reaper console overhaul (2026-06-08, session 2)

A private, offline-first knowledge-and-tools workbench: an Avalonia/.NET "pedalboard" desktop app (C:\Users\donal\ai-stack-native) over a FastAPI backend (/opt/aistack/main, on 127.0.0.1:8765) running natively in WSL2 Ubuntu — AllegroGraph, Ollama, vLLM, a Stable-Diffusion worker. This session was UI/UX: redesigning the pedal control surfaces, turning Tabby into a skill, two backend correctness fixes, a full overhaul of the Reaper engine×model console, and giving the app a real desktop identity (taskbar icon + Jump List). App commits 3be15db → d3205c7; backend commits cbf458f → 90681dd.

1. Pedal MAIN / OPTS / SYSTEM redesign — guitar-pedal control surfaces

Each pedal's sub-pages were plain Avalonia forms; they're now decisionable pedal-style controls. New partial MainWindow.Controls.cs holds a declarative ControlSpec (Knob / Switch / Selector / Dial / Button / Text) + renderers (OptKnobCtl, OptSwitchCtl, OptDialCtl, OptButtonCtl, OptTextCtl, BuildControlSurface, HeadCell) and per-tab specs OptsSpec(tab) / SysSpec(tab). One spec per content-tab drives BOTH the OPTS surface and the pedal's headline controls; every control writes an existing state field so all downstream API calls are unchanged. Mapping: numeric → knobs, fixed enums → rotary selectors, booleans → toggle switches, actions → footswitches, free text stays as pedal-framed text fields. BuildPedal renders the MODE knob + up to two headline controls (resolved from the active variant's content-tab, so the Knowledge 4-face carousel swaps them); only the owner slot of a content-tab surfaces its headline (Engine shows Chat's memory/trace, Model none).

2. Tabby — reclassified from chat engine to code-assist skill

Tabby is no longer an engine. It's a dedicated slot-6 skill: sk_tabby routes to the active Engine × Model (Ollama/vLLM × Nemotron/Twin-Llama) via localllm.generate with a code-focused prompt, tagging the reply with the engine:model that served it. The Tabby pedal's "Try in Chat" footswitch jumps to the Chat frame and prefills a coding prompt. Qwen coder weights removed (~3.5 GB); Tabby dropped from the Reaper engine axis (engines = ollama, vllm).

3. Backend correctness fixes

  • vLLM VRAM lifecycle — set_engine(vllm) now _ollama_evict()s (POST keep_alive:0 + poll

/api/ps) BEFORE starting vLLM, so its KV-cache init no longer dies with "No available memory for the cache blocks"; releasing vLLM uses _vllm_stop() (systemctl stop → SIGKILL the cgroup, since vLLM outlives SIGTERM and pins ~11.7 GB). ai-vllm unit gained KillMode=mixed / SIGINT / TimeoutStopSec=10 / FinalKillSignal=SIGKILL.

  • Memory recall fidelity — memory.recall_facts() reads the user's durable save_fact triples

from data.facts as exact lines; run_agent injects them last, labelled as outranking the fuzzy vector memory, so a self-poisoned past answer can never override a stored triple. Vector recall threshold 0.4 → 0.5.

4. Reaper Main — engine × model console overhaul (the big one)

The Reaper page is a VRAM manager + 2×2 Engine(Ollama/vLLM) × Model(Nemotron/Twin-Llama) matrix.

  • Removed the legacy Load/Release vLLM buttons — the 2×2 cells already load vLLM (vLLM cell) and

release it (Ollama cell) on engage. Action row is now Bench active / Bench all / Refresh footswitches.

  • Cell label = what the button ENGAGES, not the fallback runtime. matrix() sets resolves_to

from _engaged_label() so the non-native cells read vllm:nemotron and ollama:twin-llama (were both misleadingly ollama:nemotron); a runs_on field carries the real fallback runtime.

  • Status is relative to the ENGAGED cell — exactly one LIVE. Was static per pairing (two could

read LIVE; LIVE never moved). _cell_status(engine, model, active) via _native_up/_native_exists: LIVE (green — engaged + natively serving, only one) · READY (cyan — engine up, click to switch instantly) · LOADABLE (amber — native backend exists but its engine is stopped) · FALLBACK (muted — no native backend; runs on Ollama nemotron). Verified live: engaging vLLM×Twin → it goes LIVE and Ollama×Nemotron → READY; switching back stops vLLM so its cell → LOADABLE.

  • Realtime — the matrix + GPU telemetry are hooked into the existing 3 s poll while the Reaper

page is visible, so badges and tok/s track live state without a manual Refresh.

  • Performance line graphs — Throughput (tok/s) and First-token (TTFT ms) plotted per cell from a

rolling 30-sample bench history, colour-coded with a legend; page wrapped in a ScrollViewer.

5. App desktop identity

  • Taskbar Jump List — JumpList.cs (shell COM interop, explicit AppUserModelID) adds Tasks on

6. Ops note (recorded to memory)

ai-api (FastAPI :8765) and ai-console (:7860) are transient systemd units created by start-stack.sh via systemd-run — a bare systemctl stop ai-api DESTROYS the unit ("Unit not found" on restart). Bounce the API with sudo bash /opt/aistack/start-stack.sh (or the validated stop → reset-failed → systemd-run recreate), never systemctl restart ai-api.
