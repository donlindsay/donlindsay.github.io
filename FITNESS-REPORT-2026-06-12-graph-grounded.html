FITNESS REPORT — Graph-Grounded Generation (facts from the graph, not the prompt)

FieldEntry
SubjectThe stack's documentation & content pipeline, evaluated as graph-grounded generation
Reporting Period2026-06-12
Reporting SeniorOperator finding, relayed through the documentation workflow
BilletDevlogs and website content whose facts of record are RETRIEVED FROM AllegroGraph, not supplied in the prompt

1. Opening statement

Concur, in full, with the finding that writing a devlog should involve AllegroGraph. The facts of record for this stack live in the knowledge graph — the council seat roster, the dataset catalog, the chain of custody — and generated prose must source those facts from the graph rather than from a fact-set pasted into the prompt. Prompt-fed facts test composition; graph-retrieved facts test truth. The correction is granted, with one consequence: faithfulness is now a measurable quantity, scored against the live store.

2. Graded marks

  • (a) Grounding discipline — facts retrieved, not pasted — ★★★★☆

The writing battery now retrieves its fact-set from AllegroGraph at runtime (the seat roster via shell:invokesModel; the catalog via dcat:dataset + dcterms:description) and writes from the returned rows. One mark withheld until every authoring path — not only the battery — draws from the graph by default.

  • (b) Faithfulness is checkable — ★★★★★

Because the facts come from a query, every claim can be traced to a triple. The battery scores coverage of graph-true tokens and flags fabrications — any plausible name the graph never returned — turning "did it hallucinate?" into a deterministic check rather than a judgment call.

  • (c) End-to-end retrieve-then-write — ★★★★☆

The battery's capstone task makes the model write its own SPARQL, executes it against the graph, and feeds the rows back to be written up — the real agentic loop. Marked down only because retrieval quality varies by model and quant; that variance is exactly what the battery exists to measure.

  • (d) Prior practice — prompt-fed facts — ★★☆☆☆ (the finding / action item)

The earlier writing harness hardcoded the fact-set in the prompt. It produced fluent prose and proved nothing about truth, while the carefully-built provenance store sat unused. This is the deficiency the correction addresses.

3. Corrective actions / development plan

  • Default every authoring path to retrieval. Devlogs and site sections should open by querying the graph for the entities they describe (datasets, seats, repos, custody events), then compose from the rows — the pattern build_site.py already uses to lay out this site from a SKOS taxonomy in site.kg.
  • Make faithfulness a gate, not a vibe. Keep scoring coverage and fabrication against the graph; a page that names a thing the graph never returned fails review.
  • Publish the principle. A companion explainer — Facts Come From the Graph, Not the Prompt — now states the stance plainly, and is itself grounded in the live catalog it describes.

4. Answer to the closing question

Should the knowledge graph be in the loop when the stack writes about itself? Yes — by construction. The graph is the system of record (data.facts is authoritative; mem.agent self-poisons). Content that originates in the prompt launders unsourced claims into authoritative-looking output; content that originates in the graph carries the graph's provenance forward. Retrieve, then write.

5. Reporting Official

Prepared as a documentation-workflow assessment. Fittingly, the writing battery that enforces this report's finding is itself graph-grounded: its fact-set is retrieved from AllegroGraph and its outputs are scored against the same store — the report describes a discipline it also obeys.
